**Дата:** 2026-07-05

**Опирается на:** `docs/superpowers/specs/2026-07-04-smacznego-delivery-design.md` (общая продуктовая спека)
**Предыдущая фаза:** Ф0+Ф1 (Фундамент + Меню) — в `main`.

## 1. Обзор и цель

Превратить просматриваемое меню в **оформляемые заказы**. Клиент собирает корзину
(с выбором опций), оформляет доставку/самовывоз, и заказ создаётся в БД со
статусом `pending` (менеджер подтвердит на кухонном пульте — это Ф3). Клиент
попадает на страницу статуса своего заказа.

### Зафиксированные решения (из брейншторма Ф2)

| Вопрос | Решение |
|---|---|
| Объём корзины | Блюдо + опции (размер/добавки/«убрать»). Комбо/сеты — отдельным шагом позже |
| Форма оформления | Одна страница (mobile-first) |
| Адрес / зона | Карта + геокодинг (OpenStreetMap: Leaflet + Photon), проверка зоны по радиусу от ресторана |
| Часы работы | Блокировка вне часов + **предзаказ на время** (слот-пикер) |
| Провайдер карт | OpenStreetMap — Leaflet (карта, тайлы OSM) + Photon (автокомплит/геокодинг). Без ключа/биллинга |
| Оплата | Только при получении (наличные/терминал) — из общей спеки |
| Аккаунты | Гость; `customers` upsert по телефону; сохранённые адреса — на будущее |

### Декомпозиция на два плана

Ф2 велика, поэтому реализуется двумя планами (Ф2b разблокирован, идёт после Ф2a):

- **Ф2a — Корзина + опции + оформление (core).** Пикер опций, корзина
  (localStorage + контекст), `restaurant_settings`, одностраничное оформление с
  **ручными полями адреса** + предзаказ + валидации, серверное создание заказа с
  ревалидацией, страница заказа. Полный рабочий цикл заказа сам по себе.
- **Ф2b — Карта и геокодинг.** Заменяет ручной ввод адреса на Leaflet + Photon
  (автокомплит + перетаскиваемый пин) и радиусную проверку зоны. Оформление в Ф2a
  проектируется с **абстракцией адреса**, чтобы карта встала слоем, не переписывая поток.

### Вне рамок Ф2 (осознанно отложено)

- Комбо/сеты в корзине (отдельный небольшой шаг после Ф2a).
- Онлайн-оплата (общая спека — позже).
- Живой реалтайм на странице заказа (Ф5; здесь снапшот + опц. лёгкий поллинг).
- Кухонный пульт / подтверждение менеджером (Ф3).
- Полноценные сохранённые адреса и личный кабинет (Ф6).

## 2. Модель данных (добавления к схеме меню)

Деньги — integer **грошы**. Время — `Europe/Warsaw`.

### restaurant_settings (singleton, одна строка)
- `id` (фиксированный, напр. `true`-boolean PK или единственный uuid), `name`,
  `phone`, `address_text`.
- `lat` double, `lng` double, `delivery_radius_m` int.
- `delivery_fee` int (грошы), `free_delivery_threshold` int, `min_order` int.
- `hours` jsonb — недельное расписание: на каждый день недели `{open:"HH:MM", close:"HH:MM"}` или `null` (выходной).
- `prep_lead_minutes` int — минимальное время на готовку (для ближайшего слота предзаказа).
- RLS: публичное чтение; запись — только персонал (позже).

### orders
- `id` uuid, `public_token` (uniq, для трекинга без входа).
- `customer_name`, `customer_phone` (снапшот).
- `type` (`delivery` | `pickup`), `status` (старт `pending`; enum как в общей спеке),
  `payment_method` (`cash`), `cash_change_from` int nullable.
- `subtotal` int, `delivery_fee` int, `discount` int, `total` int (грошы) — считаются на сервере.
- `notes` text nullable.
- `address_snapshot` jsonb nullable (для delivery): `{street, building, apartment, floor, entrance, intercom, lat, lng, formatted}`.
- `scheduled_for` timestamptz nullable (`null` = «как можно скорее»).
- `language` (`pl`|`uk`|`ru`), `created_at`, статус-таймстемпы.
- `customer_id` uuid nullable (fk customers).

### order_items
- `order_id` fk, `dish_id` nullable (ссылка-справка), **снапшот** `name` (текст на языке заказа),
  `unit_price` int, `qty` int, `selected_options` jsonb (снапшот: массив `{group, name, price_delta}`),
  `line_total` int.

### customers
- `id` uuid, `phone` (unique), `name`, `created_at`. Upsert по телефону при заказе.

### addresses (минимально, для будущего переиспользования)
- `id`, `customer_id` fk, поля адреса + `lat`/`lng`, `label` nullable. В Ф2 адрес и так
  лежит в `orders.address_snapshot`; таблица заполняется, но UI выбора сохранённого — позже.

## 3. Корзина и пикер опций (клиент)

- **Хранение:** гостевая корзина в `localStorage`, доступ через React-контекст
  (`CartProvider`). Пункт корзины = `{dishId, name-snapshot, unitPrice, qty, selectedOptions[]}`.
- **Пикер опций:** клик по блюду с опциями открывает нижний лист/модал: обязательный
  «Размер» (ровно 1 из N), «Добавки» (0..N), «Убрать» (0..N), счётчик количества,
  живая цена. Кнопка «Добавить» кладёт снапшот выбора + вычисленную цену в корзину.
  Блюдо без опций добавляется в один клик.
- **Пересчёт (чистые функции):** `subtotal`, `deliveryFee` (0 при `subtotal ≥ free_delivery_threshold`),
  индикатор «до бесплатной доставки осталось N zł», флаг недобора `min_order`.
- Иконки — только SVG (без эмодзи), тёмный гурмэ, токены `@theme`.

## 4. Оформление (одна страница)

Разделы одного экрана:
1. **Способ:** тумблер доставка/самовывоз.
2. **Контакт:** имя, телефон (валидация польского формата, нормализация).
3. **Адрес** (только delivery): Ф2a — поля (улица/дом/кв/этаж/подъезд/домофон);
   Ф2b — карта+автокомплит поверх той же абстракции. Поля кв/этаж/подъезд/домофон/коммент — всегда вручную.
4. **Время:** «как можно скорее» или слот из `generateSlots(hours, prep_lead_minutes, now)`.
5. **Комментарий** к заказу; при наличных — «с какой суммы сдача?» (опц.).
6. **Сводка:** позиции, subtotal, доставка, total; блок кнопки при недоборе мин. заказа.
7. **«Заказать».** Если закрыто и слотов нет — состояние «открыто с HH:MM».

## 5. Создание заказа (server action)

Клиенту не доверяем:
1. Заново читаем блюда/опции из БД по id из корзины.
2. Проверяем `is_available` (стоп-лист) каждой позиции → иначе ошибка «блюдо закончилось».
3. Пересчитываем `unit_price`/`line_total`/`subtotal`/`delivery_fee`/`total` **на сервере**
   (клиентские суммы игнорируем).
4. Проверяем `subtotal ≥ min_order`.
5. Проверяем время: открыто сейчас **или** валидный `scheduled_for` слот (часы + `prep_lead_minutes`).
6. Для delivery — адрес в зоне: `haversine((lat,lng), restaurant) ≤ delivery_radius_m` (Ф2b; в Ф2a — заглушка/пропуск до карты).
7. Upsert `customers` по телефону, создаём `orders`(`pending`) + `order_items` (снапшоты), генерируем `public_token`.
8. Защита от дублей (двойной клик / повтор): идемпотентность по короткому окну (напр. одинаковый телефон+сумма+позиции за N секунд).
9. Возвращаем токен → редирект на `/order/[token]`.

## 6. Страница заказа `/order/[token]`

Серверный рендер: статус (`pending` → «ждёт подтверждения менеджера»), позиции,
итоги, адрес/время, контакт, способ оплаты. Токен = единственный «ключ» (без входа).
**Живой реалтайм — Ф5**; здесь снапшот, опционально лёгкий поллинг статуса.

## 7. Карта и геокодинг (Ф2b)

- **Leaflet** + тайлы OSM (обязательная атрибуция «© OpenStreetMap contributors»).
- **Photon** (`photon.komoot.io`) для автокомплита адреса: дебаунс ~300 мс, ограничение
  bbox/приоритет вокруг города ресторана, «fair use» (не молотить запросами).
- Перетаскиваемый пин → `lat/lng` + форматированный адрес (reverse geocode).
- Поля кв/этаж/подъезд/домофон/коммент — вручную поверх координат.
- Серверная проверка зоны по радиусу; вне зоны — блок с сообщением.
- Абстракция: компонент `AddressInput` с единым выходом `{fields, lat, lng, formatted}`,
  Ф2a — реализация на полях, Ф2b — на карте; server action и модель не меняются.

## 8. Подводные камни

1. Стоп-лист/цена изменились между добавлением и оформлением → ревалидация на сервере, понятное сообщение.
2. Таймзона `Europe/Warsaw` для часов/слотов; хранение — timestamptz.
3. Предзаказ в закрытый день/время → слот-пикер отдаёт только валидные слоты; сервер перепроверяет.
4. Двойной клик «Заказать» → идемпотентность/анти-дубль на server action.
5. Деньги — целые грошы, без float; все итоги считаются на сервере.
6. Photon/OSM «fair use» → дебаунс, кэш последнего результата, атрибуция.
7. Пустая корзина / прямой заход на `/checkout` без позиций → редирект в меню.
8. `restaurant_settings` отсутствует/не сконфигурирован → безопасные дефолты + явная ошибка админу (сидируем строку).

## 9. Тестирование

- **Юнит:** расчёт корзины (subtotal/доставка/скидка/до-бесплатной), `generateSlots` из часов+lead, `haversine`/радиус, валидация телефона, серверная ревалидация цен.
- **Интеграция:** создание заказа — happy path + стоп-лист + недобор мин. заказа + вне часов + вне зоны + анти-дубль.
- **E2E (Playwright):** блюдо с опциями → корзина → оформление (самовывоз, ASAP) → страница заказа; и вариант доставки с адресом/слотом.

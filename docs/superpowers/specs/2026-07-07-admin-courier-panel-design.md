**Дата:** 2026-07-07
**Статус:** Утверждён (черновик спеки, ждёт финальной вычитки)
**Опирается на:** `docs/superpowers/specs/2026-07-04-smacznego-delivery-design.md` (общая продуктовая спека), `docs/superpowers/specs/2026-07-05-cart-checkout-design.md` (Ф2 — корзина/оформление)

## 1. Обзор и цель

Достроить недостающее звено после оформления заказа: **рабочие места персонала**.
Сейчас заказ создаётся в БД со статусом `pending` и просто там лежит — нет ни
экрана для кухни/оператора (принять, готовить, выдать), ни экрана для курьера
(забрать готовый заказ, доставить). Единственный доступ — сырой Supabase Studio.

Фаза 3 добавляет два внутренних экрана на **русском** языке, вне `[locale]`-дерева,
под простым парольным гейтом:

- **`/admin`** — оператор (кухня/менеджер): все заказы за сегодня, продвижение по
  статусам жизненного цикла.
- **`/courier`** — курьер: заказы на доставку, «забрал → доставил».

Обновления — **мгновенные** (Supabase Realtime через серверный SSE) + звук на новый
заказ.

### Зафиксированные решения (из брейншторма Ф3)

| Вопрос | Решение |
|---|---|
| Вход | Простой пароль/PIN. Два кода в env: персонал → `/admin`, курьер → `/courier`. Сессия в подписанной httpOnly-cookie. Без Supabase Auth и переделки RLS |
| Свежесть списка | Мгновенно — Supabase Realtime, проксированный через серверный gated-SSE (заказы приватны, клиент не получает прямой доступ к БД). + звук на новый заказ. Фолбэк-опрос ~15 с |
| Язык UI | Русский, один язык (без i18n для этих экранов) |
| Экраны | Раздельные: `/admin` (оператор) и `/courier` (курьер), каждый со своим кодом |
| Модель курьера | Общая очередь готовых к доставке заказов (без привязки к конкретному курьеру) |
| Статусы | Переиспользуем существующий enum из `0002_orders.sql` |

## 2. Авторизация (простой пароль)

Никакой авторизации в проекте сейчас нет. Вводим лёгкий парольный гейт — без
Supabase Auth, чтобы не переделывать RLS.

**Env-переменные** (`.env.local`, `.env.example`):
- `ADMIN_PASSCODE` — код входа оператора
- `COURIER_PASSCODE` — код входа курьера
- `AUTH_SECRET` — секрет для HMAC-подписи cookie

**Поток входа:**
1. `/admin` и `/courier` без валидной cookie показывают форму входа (поле кода + кнопка).
2. Server action `login(role, passcode)` сверяет код с соответствующей env-переменной
   (сравнение в постоянном времени). При успехе ставит httpOnly, `sameSite=lax`,
   `secure` (в проде) cookie `sc_session`.
3. Значение cookie = `role|expiresAt|HMAC(role|expiresAt, AUTH_SECRET)`. Срок ~30 дней
   (устройства персонала доверенные). Проверка — пересчёт HMAC + срок.
4. Выход — server action, удаляющий cookie.

**Роли:** `staff` (доступ к `/admin`), `courier` (доступ к `/courier`). `staff`
считается надмножеством для чтения, но по умолчанию каждый код открывает свой экран.

**Гейт (middleware):** текущий `middleware.ts` — next-intl со всеохватным matcher'ом,
который редиректил бы `/admin` → `/pl/admin`. Композируем:
- Пути `^/(admin|courier)` — обрабатываем сами: проверяем cookie/роль → нет → редирект
  на форму входа этого экрана; исключаем их из intl.
- Всё остальное — как сейчас, через `createMiddleware` next-intl.
- SSE-роут под `/api/*` уже исключён текущим matcher'ом; гейт по cookie делаем в самом
  роуте.

**Защита мутаций:** каждый server action смены статуса заново читает и проверяет
cookie-сессию и роль на сервере (middleware — не единственная линия обороны).

## 3. Статусы и действия

Переиспользуем существующий CHECK-enum `orders.status`:
`pending, confirmed, preparing, ready, out_for_delivery, delivered, picked_up, cancelled, rejected`.

**Карта допустимых переходов** (единый источник в `lib/orders/statusFlow.ts`):

```
pending          → confirmed | rejected
confirmed        → preparing | cancelled
preparing        → ready | cancelled
ready (delivery) → out_for_delivery | cancelled
ready (pickup)   → picked_up | cancelled
out_for_delivery → delivered | cancelled
delivered        → (терминальный)
picked_up        → (терминальный)
cancelled        → (терминальный)
rejected         → (терминальный)
```

Server action отклоняет любой переход не по карте (и учитывает `type` заказа:
`out_for_delivery`/`delivered` — только для `delivery`; `picked_up` — только для `pickup`).

### Оператор `/admin` — доска по колонкам

| Колонка | Статус(ы) | Кнопки на карточке |
|---|---|---|
| Новые | `pending` | Принять → `confirmed` · Отклонить → `rejected` |
| Приняты | `confirmed` | Готовим → `preparing` · Отменить |
| Готовим | `preparing` | Готов → `ready` · Отменить |
| Готовы | `ready` | самовывоз: Выдан → `picked_up` · доставка: «ждёт курьера» (действий нет) · Отменить |
| В доставке | `out_for_delivery` | (обрабатывает курьер; оператор видит) |
| Завершены | `delivered`, `picked_up`, `cancelled`, `rejected` | свёрнуто/фильтр |

Карточка заказа: номер (`#` + первые 8 символов токена), тип (доставка/самовывоз),
время создания и `scheduled_for` (если предзаказ), позиции с опциями, клиент (имя,
телефон), адрес (для доставки), оплата (наличные + «сдача с»), суммы. По умолчанию —
заказы за сегодня; активные вверху.

### Курьер `/courier` — общая очередь

Два раздела:
- **К доставке** = заказы `type=delivery, status=ready` → кнопка **Забрал** → `out_for_delivery`.
- **В пути** = `out_for_delivery` → кнопка **Доставлен** → `delivered`.

Карточка курьера (крупная, mobile-first): адрес (`address_snapshot.formatted`),
телефон (tap-to-call `tel:`), сумма к оплате, «сдача с» (если задано), краткий состав.
Без привязки к конкретному курьеру — любой курьер берёт любой готовый заказ (YAGNI
для одного ресторана; назначение — на будущее).

## 4. Realtime

- **Миграция** `0004_realtime_orders.sql`: `alter publication supabase_realtime add table orders;`
  (плюс `order_items` при необходимости для деталей). Публикация `supabase_realtime`
  создаётся Supabase по умолчанию.
- **SSE-роут** `app/api/admin/stream/route.ts`:
  - Гейт: проверка cookie-сессии; без валидной роли → 401.
  - На сервере создаёт **service-role** Supabase-клиент и подписывается на
    `postgres_changes` (INSERT/UPDATE) таблицы `orders`. Service-role обязателен: `orders`
    приватна (RLS без политик/грантов anon), а Realtime `postgres_changes` уважает RLS —
    anon-подписка не получила бы событий.
  - Стримит клиенту события (`event: order`, полезная нагрузка — минимальная: id + новый
    статус/тип, или сигнал «обнови»).
  - Корректно закрывает подписку при разрыве соединения.
- **Клиент:** хук `useOrderStream(role)` подписывается на SSE; на событие вызывает
  `router.refresh()` (перечитывает серверный список). Звук: на новый `pending` (оператор)
  и на новый `ready` доставки (курьер). Фолбэк — `setInterval` опрос ~15 с, если SSE не
  подключился/отвалился.
- Звук — короткий встроенный ассет; воспроизводится после первого взаимодействия
  пользователя (ограничение автоплея браузера) — кнопка «включить звук» при входе.

## 5. Файлы / слои

**Новое:**
- `app/admin/page.tsx` — доска оператора (server component: грузит заказы за сегодня).
- `app/admin/login/page.tsx` + форма (client) → server action `login('staff', code)`.
- `app/courier/page.tsx` — очередь курьера (server component).
- `app/courier/login/page.tsx` + форма → `login('courier', code)`.
- `app/api/admin/stream/route.ts` — SSE.
- `lib/auth/passcode.ts` — HMAC подпись/проверка значения cookie; `verifyPasscode`.
- `lib/auth/session.ts` — чтение/установка/очистка cookie, `getSession()` (роль), server actions `login`/`logout`.
- `lib/orders/adminOrders.ts` — запросы (за сегодня, по статусам, курьерская очередь) через `createAdminClient`.
- `lib/orders/statusFlow.ts` — карта переходов + `canTransition(order, next)`.
- `lib/orders/updateStatus.ts` — server action: проверка роли + валидность перехода → update.
- `components/admin/*` — доска, колонка, карточка заказа, кнопки статусов.
- `components/courier/*` — карточка доставки, кнопки.
- `components/admin/OrderStream.tsx` — клиентский хук/компонент SSE + звук.

**Изменяем:**
- `middleware.ts` — композиция парольного гейта + next-intl.
- `.env.local`, `.env.example` — новые переменные.
- `supabase/migrations/0004_realtime_orders.sql` — публикация Realtime.

**Переиспользуем:** `lib/supabase/admin.ts` (`createAdminClient`), `lib/menu/price.ts`
(`formatZloty`), типы `lib/supabase/types.ts`, иконки `components/Icon.tsx`, палитру/шрифты
из `globals.css`.

## 6. Тесты (vitest, как в проекте)

- **Юнит:**
  - `statusFlow` — допустимые/запрещённые переходы, учёт `type` (delivery vs pickup).
  - `passcode` — подпись/проверка cookie (HMAC), протухший срок, подделка отклоняется.
  - `adminOrders` — фильтр курьерской очереди (только `ready` доставка / `out_for_delivery`).
- **Компонентные:**
  - Доска оператора рендерит колонки и раскладывает заказы по статусам.
  - Кнопки карточки (оператор/курьер) вызывают server action с правильным целевым статусом.
- SSE-стрим и middleware-гейт проверяем вручную при прогоне приложения (playwright/браузер),
  как «прямую часть» в Ф2.

## 7. Вне scope (YAGNI)

- Именные аккаунты курьеров и назначение заказа конкретному курьеру.
- Web-push / браузерные уведомления вне вкладки.
- Печать чеков/кухонных тикетов.
- Аналитика, отчёты, история/поиск за пределами «сегодня».
- Редактирование состава/сумм заказа оператором.
- Мультиресторан, роли тоньше, чем staff/courier.

## 8. Порядок реализации (черновик, детализируется в плане)

1. Парольный гейт: env, `lib/auth/*`, middleware, формы входа. (Проверяемо: `/admin` требует код.)
2. Данные + статусы: `statusFlow`, `adminOrders`, `updateStatus` + юнит-тесты.
3. Экран оператора `/admin`: доска, карточки, кнопки статусов.
4. Экран курьера `/courier`: очередь, карточки, действия.
5. Realtime: миграция публикации, SSE-роут, клиентский хук + звук, фолбэк-опрос.
6. Прогон «прямой части» end-to-end: заказ клиента → оператор принимает/готовит → курьер доставляет.

# Smacznego — сайт ресторана с доставкой. Дизайн-спека

**Дата:** 2026-07-04
**Статус:** Утверждён (черновик спеки, ждёт финальной вычитки)
**Тип:** Продуктовая спека для одного ресторана (реальный бизнес, Польша)

---

## 1. Обзор и цель

Веб-приложение для **одного конкретного ресторана** (реальный бизнес в Польше),
позволяющее клиентам заказывать доставку/самовывоз, а ресторану — принимать и
вести эти заказы в реальном времени. Приоритеты: **простота, минимализм и
по-настоящему крутой, «дорогой» визуал**.

Это не «сайт», а три взаимосвязанных интерфейса по ролям:

1. **Клиент** — смотрит меню, собирает корзину, оформляет заказ, следит за статусом.
2. **Кухня/менеджер** — видит входящие заказы мгновенно, подтверждает, ведёт по статусам.
3. **Курьер** — видит свободные заказы, «забирает» их, отмечает доставку.

### Зафиксированные решения (из брейншторма)

| Вопрос | Решение |
|---|---|
| Масштаб | Один ресторан (не агрегатор) |
| Стадия | Реальный бизнес |
| Оплата | Только при получении (наличные/терминал курьеру). Онлайн-оплата — НЕ в MVP, но модель это учитывает |
| Доставка | Своя доставка + самовывоз |
| Аккаунты | Гость + опциональный аккаунт по телефону (история/адреса) |
| Языки | Польский (по умолчанию) + украинский + русский |
| Фейки/статусы | Без SMS на старте: менеджер подтверждает звонком + живой трекинг на странице заказа. SMS-верификация заложена как включаемая на будущее |
| Курьеры | Самозабор из общего пула (атомарный claim, первый выигрывает) |
| Доставка (цена) | Фикс-стоимость + бесплатно от суммы + минимальный заказ (всё настраиваемо) |
| Меню | Категории → блюда → опции (размер/добавки/«без …») + комбо/сеты + акции + стоп-лист |
| Визуал | Тёмный гурмэ: эспрессо + золото, серифный дисплейный шрифт, **только SVG-иконки, без эмодзи, реальные фото блюд** |
| Стек | Next.js (App Router, TS) + Supabase (Postgres, Realtime, Auth, Storage), Tailwind + shadcn/ui, next-intl |

### Вне рамок MVP (осознанно отложено)

- Онлайн-оплата (карта/BLIK/Przelewy24) — модель данных совместима, добавляется позже.
- Полные SMS-уведомления на каждый статус и SMS-верификация номера.
- Автоназначение/оптимизация маршрутов курьеров.
- Мультиресторанность/агрегатор.
- Предзаказ на конкретное время (scheduled) — кандидат на доработку после MVP.
- Учёт остатков по количеству (только вкл/выкл стоп-лист).

## 2. Архитектура и роли

Одно Next.js-приложение (App Router, TypeScript) с зонами по ролям. Supabase —
бэкенд: **Postgres** (данные), **Realtime** (вебсокеты), **Auth** (персонал),
**Storage** (фото блюд). Регион — **EU (Франкфурт)** ради RODO/GDPR.

- **Клиент** — `/` и подмаршруты, публично, **mobile-first** (там реально заказывают).
  Меню, корзина, оформление, `/order/[token]` — трекинг.
- **Кухня/менеджер** — `/kitchen`, вход по логину, **desktop-first** (большой экран на кухне).
- **Курьер** — `/courier`, вход, **mobile**.

Сквозные правила:
- Деньги — **integer в грошах**, без float.
- Время — таймзона `Europe/Warsaw`.
- Доступ к данным — серверные экшены / route handlers; реалтайм — Supabase-подписки.
- Ролевой доступ к `/kitchen` и `/courier` — через Supabase Auth + middleware.

## 3. Жизненный цикл заказа

Ядро системы — конечный автомат статусов.

```
pending ──(менеджер подтвердил звонком)──▶ confirmed ──▶ preparing ──▶ ready
                                                                          │
                                        delivery: ready ─▶ out_for_delivery ─▶ delivered
                                        pickup:   ready ─▶ picked_up
   любой ранний этап ──▶ cancelled / rejected (с причиной)
```

Правила переходов:
- **pending → confirmed**: только менеджер. До `confirmed` заказ на кухню НЕ уходит
  (фильтр от фейков через подтверждающий звонок).
- **Курьерский пул** = заказы со статусом `ready` **и** `type = delivery` **и**
  `courier_id IS NULL`.
- **Атомарный claim**: `UPDATE orders SET courier_id = :me, status = 'out_for_delivery'
  WHERE id = :id AND courier_id IS NULL`. Если затронуто 0 строк — «заказ уже взят».
- **Самовывоз**: `ready → picked_up` (курьер не участвует).
- **Отмена/отклонение**: доступны на ранних этапах, всегда с `cancel_reason`.
- Каждый переход фиксируется в `order_events` (аудит и история).

Статусы: `pending | confirmed | preparing | ready | out_for_delivery |
delivered | picked_up | cancelled | rejected`.

## 4. Модель данных

Ключевые сущности (детали типов уточняются на этапе плана реализации).

### Меню
- **categories** — i18n-имя (pl/uk/ru), порядок сортировки, видимость.
- **dishes** — i18n имя/описание, `base_price` (грошы), `category_id`, `photo_url`,
  `is_available` (стоп-лист), теги (веган/острое/…), сортировка.
- **option_groups** — привязка к блюду; i18n-имя; `min_select`/`max_select`;
  `required`. Примеры: «Размер» (required, 1 из N), «Добавки» (0..N), «Убрать» (0..N).
- **options** — `option_group_id`, i18n-имя, `price_delta` (грошы, может быть 0).
- **combos** — i18n имя/описание, цена, фото; **combo_items** — состав (ссылки на блюда/категории).
- **promos** (опц.) — код, тип и величина скидки, условия применения.

### Заказы
- **orders** — `public_token` (трекинг без входа), снапшот контактов (имя/телефон),
  `type` (delivery|pickup), `status`, `payment_method` (cash|terminal),
  `cash_change_from` (с какой суммы сдача, опц.), суммы
  (`subtotal`/`delivery_fee`/`discount`/`total`), `notes`, `address_snapshot`,
  `courier_id`, таймстемпы по статусам, `language`, `created_at`.
- **order_items** — **снапшот** имени/цены блюда и выбранных опций на момент заказа
  (правки меню не меняют прошлые заказы), `qty`, `line_total`.

### Люди
- **customers** — `phone` (unique), имя, опциональный аккаунт (история/адреса).
  Создаётся при первом заказе.
- **addresses** — `customer_id`, улица/дом/квартира/этаж/подъезд/домофон/заметки,
  опц. гео (lat/lng).
- **staff** — `role` (manager|kitchen|courier), имя, через Supabase Auth; у курьера
  статус online/offline.

### Настройки
- **restaurant_settings** — часы работы, `delivery_fee`, `free_delivery_threshold`,
  `min_order`, зоны доставки (список индексов/радиус), телефон, адрес. Редактируемо, не хардкод.

### Аудит
- **order_events** — `order_id`, `from_status`, `to_status`, `actor`, `timestamp`.

## 5. Реалтайм-потоки

- **Кухня/менеджер** — подписка на активные `orders` за сегодня. Новый `pending`:
  карточка появляется + **звук** + подсветка «не подтверждён» + счётчик непринятых.
  Канбан-колонки: Новые → Готовится → Готов → В пути → Завершён. Старые `pending`
  визуально эскалируются (чтобы не пропустить).
- **Курьер** — подписка на пул `ready`-доставок + свой активный заказ. Кнопка
  «Забрать» (атомарно) → «Доставлен».
- **Клиент** — `/order/[token]` подписан на свой заказ; статусы обновляются вживую
  (главный «вау»-элемент вместо тишины).

## 6. Подводные камни и решения

1. **Фейковые заказы** → gate «менеджер подтвердил» до кухни; SMS-верификация как включаемая опция на будущее.
2. **Гонка курьеров за один заказ** → атомарный conditional update.
3. **Правки меню vs заказы в работе** → снапшот позиций в `order_items`.
4. **Стоп-лист в час пик** → `is_available=false` прячет блюдо; повторная валидация корзины при оформлении («блюдо закончилось»).
5. **Ресторан закрыт / вне часов** → оформление заблокировано, показываем «открыто с …».
6. **Мин. заказ не набран / адрес вне зоны** → блок оформления с понятным сообщением; зона по индексу/списку + финальное решение менеджера.
7. **Сдача с наличных** → поле «с какой суммы сдача?».
8. **Пропущенный заказ на кухне** → звук + подсветка + счётчик; эскалация старых `pending`.
9. **RODO/GDPR** → минимум PII, EU-регион, политика приватности, срок хранения.
10. **Мультиязычность** → переводы меню; фолбэк на польский при пустом переводе.
11. **Деньги/точность** → integer грошы, без float.
12. **Таймзоны** → `Europe/Warsaw` везде.
13. **Дубли/спам заказов** → дедуп по phone+время, базовый rate-limit.

## 7. Дизайн-система

- **Направление:** тёмный гурмэ. Премиально, аппетитно, минималистично.
- **Цвета:** эспрессо-фон `#14100d` / `#1c1611`; золото `#e0a44a`; кремовый текст
  `#f6ecdd`; приглушённый `#b39d84`; success-зелёный; danger-красный.
- **Типографика:** серифный дисплейный шрифт (напр. Fraunces / Cormorant) для
  заголовков + Inter для текста.
- **Иконки/медиа:** **только SVG-иконки, никаких эмодзи**, реальные оптимизированные
  фото блюд (Supabase Storage + Next Image).
- **Инструменты:** Tailwind CSS + shadcn/ui (для админки). Клиент — mobile-first;
  кухня — desktop-first; курьер — mobile.
- Референс-макеты сохранены в `.superpowers/brainstorm/` (тёмный гурмэ, v2 на SVG).

## 8. План сборки по фазам

Проект большой (3 интерфейса), поэтому спека описывает продукт целиком, а
реализацию бьём на фазы — каждая со своим планом. **Ф1–Ф3 = основная петля**
(заказ доезжает до кухни), с неё начинаем.

- **Ф0 — Каркас.** Next.js + Tailwind + Supabase + next-intl + дизайн-токены; ролевые маршруты, middleware.
- **Ф1 — Меню.** Модель данных меню + админ-CRUD (или сид) + публичный просмотр (3 языка, категории, опции, стоп-лист).
- **Ф2 — Корзина + оформление.** Гость, адрес, доставка/самовывоз, расчёт комиссии и мин. заказа, валидация → создание заказа.
- **Ф3 — Пульт кухни/менеджера.** Реалтайм-канбан, подтверждение, статусы, звук.
- **Ф4 — Экран курьера.** Пул, атомарный claim, «в пути/доставлен».
- **Ф5 — Трекинг клиента.** `/order/[token]`, живой статус.
- **Ф6 — Довесок.** Опц. аккаунты, RODO-страницы, полировка, подпись автора в футере.

## 9. Тестирование

- **Юнит** — расчёт цен/комиссии/скидок, логика claim, валидация корзины.
- **Интеграция** — жизненный цикл заказа, реалтайм-подписки.
- **E2E (Playwright)** — happy path заказа клиентом + приём на кухне.
- **Конкурентность** — отдельный тест на гонку claim (двое курьеров, один заказ).
